# Hybrid-mode H2a micro-benchmark — lwIP TCP termination gate

**Date:** 2026-07-03
**Gate (spec H2a):** classifier + lwIP TCP termination must sustain **>= 10 Gbit/s** single-flow at MTU 8500, with no QUIC in the path.
**Verdict: PASS** — 10.47 Gbit/s (Debug build), 21.79 Gbit/s (Release build).

## 1. What is measured

The H2a Definition-of-Done gate for hybrid-mode Chunk 1: raw throughput of
the TCP-lane ingress path in isolation — TUN read → `mqvpn_hybrid_classify()`
→ `mqvpn_lwip_input()` → lwIP wildcard-intercept TCP termination →
byte sink. No QUIC, no relay, no scheduler. This bounds what the full
TCP lane can ever deliver and catches config/integration bugs (checksum
misconfig, pbuf copies, MTU/MSS mismatch) before the QUIC side lands.

Rig: `benchmarks/tcp_lane_microbench/harness.c` — a standalone throwaway
executable linking `lwip_core` + `src/hybrid/classifier.c` +
`src/hybrid/lwip_glue.c` directly (CMake target `tcp_lane_microbench`,
built only under `-DMQVPN_ENABLE_HYBRID_TCP_LANE=ON`). Driver:
`benchmarks/tcp_lane_microbench/run.sh`.

## 2. Topology and methodology

Single isolated network namespace, loopback-through-TUN topology:

```
python bulk sender ──connect 10.99.0.2:5201──▶ kernel TCP (cubic)
        ▲                                          │ route 10.99.0.0/24 → tunbench0 (MTU 8500)
        │ ACKs / SYN-ACK                           ▼
kernel ◀──write(tun_fd)── harness: read(tun_fd) → classify → lwIP input
                                   └─ accepted-connection bytes → sink (counted, freed)
```

- The harness creates `tunbench0` (TUNSETIFF, `IFF_TUN|IFF_NO_PI`) and
  self-configures 10.99.0.1/24 + MTU 8500 via ioctl (no iproute2 needed
  in-namespace). The kernel's connected route sends 10.99.0.2 into the TUN.
- lwIP's pretend-netif wildcard listener (heiher fork intercept) accepts the
  SYN for 10.99.0.2:5201; handshake/ACK segments generated by lwIP are
  written back into the same TUN fd (`netif->output` glue path).
- Measurement is **receiver-side**: bytes accepted by the lwIP recv callback
  between the flow's first and last payload byte (harness `RESULT` line).
  Sender-side and receiver-side byte counts matched exactly in every run.
- Single flow, 10 s, effective MSS 8460 (netif MTU 8500 − 40), lwIP window
  ~2 MB (`TCP_WND = 65535 << 5`), checksums ON both directions
  (`CHECKSUM_CHECK_TCP/GEN_TCP = 1`, the v1 fuzz-safety default — not
  flipped for the benchmark).

### 2.1 Execution environment (rootless caveat)

`sudo` was not available in this session and
`kernel.apparmor_restrict_unprivileged_userns=1` blocks `unshare -r -n` on
this host, so the official `sudo bash run.sh` (ip-netns mode) could not be
run. The measurement ran in a **NET_ADMIN docker container** instead — same
kernel, no virtualization overhead, an isolated netns exactly like the
ip-netns flow:

```
# image: ubuntu:24.04 + python3, all run.sh needs in-container:
#   FROM ubuntu:24.04
#   RUN apt-get update && apt-get install -y --no-install-recommends python3
# (the actual bench image also carried iperf3/iproute2, used only for the
#  §2.2 dry-run diagnostics)
docker run --rm --network none --cap-add NET_ADMIN --device /dev/net/tun \
  -v <repo>:<repo>:ro -e MQVPN_BENCH_IN_NS=1 <image> \
  bash <repo>/benchmarks/tcp_lane_microbench/run.sh
```

The committed `run.sh` executed **verbatim** inside the container
(`MQVPN_BENCH_IN_NS=1` entry mode); host glibc (2.39) matches ubuntu:24.04,
so the CMake-built binaries ran unmodified. An official `sudo bash run.sh`
re-run on the bare host remains as a user checkpoint; no material
difference is expected (the container shares the host kernel and CPU).

Host: AMD Ryzen 9 5950X (32 threads), Linux 6.17.0-29-generic.

### 2.2 iperf3 substitution (dry-run finding)

The plan's sketch drove traffic with `iperf3 -s` + `iperf3 -c`. The dry run
confirmed both halves cannot work against a termination sink:

- `iperf3 -s` is **dead weight**: the SYN to 10.99.0.2:5201 enters the TUN
  and is terminated by lwIP's wildcard listener — no packet can ever reach a
  real server socket.
- `iperf3 -c` **stalls**: it completed the TCP handshake against lwIP and
  sent only its control-channel opener — the harness sank exactly 38 bytes,
  the 37-byte cookie plus one control byte — then blocked forever waiting
  for the server's `PARAM_EXCHANGE` state byte, which a pure sink never
  sends.
  The client died only via `timeout` (exit 124) after transferring nothing.

`run.sh` therefore uses a plain python bulk sender (blocking `send()` of
1 MiB buffers for 10 s, then `shutdown(SHUT_WR)`), and the gate metric is
the harness's receiver-side rate. The metric itself is unchanged from the
spec: sustained bytes/sec through classifier + lwIP termination at MTU 8500.

### 2.3 Tick-loop note

The harness calls `mqvpn_lwip_tick()` once per TUN `read()`. Under load
reads dominate (>100 k/s), so the 250 ms `tcp_tmr` cadence is trivially met;
when idle the loop blocks in `read()` and timers stall. This is fine for the
benchmark (no retransmit work exists while no packets flow) and does not
distort the measurement; the real integration drives ticks off
`mqvpn_lwip_next_timeout_ms()`.

## 3. Results

|---|---|---|---|

| CMake Release (scratch dir, same config) | `-O3`, `LWIP_NOASSERT` | **21.79 Gbit/s** | PASS |
| manual static `-O0 -g` (cross-check) | | 10.58 Gbit/s | PASS |

Per-second progress was stable (Release: 20.2–22.5 Gbit/s; Debug:

## 4. Comparison to reference points

| Stack | Throughput | Note |
|---|---|---|
| hev-socks5-tunnel (same heiher/lwip base) | 32.8 Gbit/s @ MTU 8500 | upstream reference |
| **mqvpn H2a rig (this report, Release)** | **21.79 Gbit/s** | checksums ON, 2 copies/packet (read buf → pbuf_take) |
| **mqvpn H2a rig (this report, Debug)** | **10.47 Gbit/s** | asserts + `-g`, no optimization |
| badvpn (untuned) | 2.9 Gbit/s | "landing near 3 = config bug" tripwire |

The Release number sits at ~66 % of hev-socks5-tunnel on the same lwIP
base — consistent with this rig's deliberate simplicity (single-threaded
read loop, full TCP checksum verify+generate per the v1 fuzz-safety
default, an extra `pbuf_take` copy on ingress and `pbuf_copy_partial` on
egress). Nowhere near the badvpn ~3 Gbit/s misconfiguration signature; MSS
derivation from `netif->mtu` (8460) and the 2 MB scaled window were
confirmed working by the stable per-second rate.

## 5. Gate decision

**PASS.** Both build flavors clear 10 Gbit/s — the Debug build with only
~5 % headroom (expectedly: `-O0` + lwIP asserts), the Release build with
2.2x. The H2a spec point — lwIP + classifier termination is not the
bottleneck relative to the ~3–5 Gbit/s QUIC path budget — holds with wide
margin.
